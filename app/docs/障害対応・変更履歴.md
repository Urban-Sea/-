# 障害対応・変更履歴

> 運用中に発生した問題と対応を記録するドキュメント

---

## 2026-02-26: 保有データがユーザー間で共有されていた

### 問題

Cloudflare Access で公開許可した友人のアカウントからログインすると、**自分の保有タブ（holdings）のデータがそのまま見えていた**。本来はユーザーごとにデータが分離されるべきだが、全員が同じデータを閲覧・編集できる状態だった。

### 原因（2つ）

#### 1. フロントエンドがAPIリクエストにユーザー識別ヘッダーを送信していなかった

- Cloudflare Access は `open-regime.pages.dev` へのアクセスゲートとして機能し、ログイン済みユーザーのメールは `/cdn-cgi/access/get-identity` で取得可能
- しかし、フロントエンドの `fetchAPI()` / `swrFetcher()` は API リクエスト時に `CF-Access-Authenticated-User-Email` ヘッダーを**付与していなかった**
- Worker（`open-regime-api`）はフロントエンドとは別ドメインのため、Cloudflare Access のヘッダー自動付与が効かない

#### 2. Railway の `ENVIRONMENT` が `production` に設定されていなかった

- バックエンドの `auth.py` は `ENVIRONMENT != "production"` の場合、認証ヘッダーがなくても `dev@localhost` をデフォルトユーザーとして返す
- 結果、**全ユーザーが `dev@localhost` として識別**され、同じ保有データを共有

```
全ユーザー → ヘッダーなし → auth.py → dev@localhost → 全員同じデータ
```

### 対応

#### フロントエンド修正

| ファイル | 変更内容 |
|----------|----------|
| `app/frontend/src/lib/auth-store.ts` | 新規作成。モジュールレベルでメールを保持するストア |
| `app/frontend/src/components/providers/UserProvider.tsx` | Cloudflare Access から取得したメールを auth-store にセット。メール取得後に `mutate(() => true)` で全SWRキーを再検証 |
| `app/frontend/src/lib/api.ts` | `fetchAPI()` に `CF-Access-Authenticated-User-Email` ヘッダーを付与 |
| `app/frontend/src/lib/swr.tsx` | `swrFetcher()` に同ヘッダーを付与 |
| `app/frontend/src/app/layout.tsx` | `SWRProvider` を `UserProvider` の外側に移動（UserProvider 内で `useSWRConfig` を使えるように） |

#### 認証フロー（修正後）

```
ブラウザ → Cloudflare Access ログイン
  → /cdn-cgi/access/get-identity でメール取得
  → auth-store にセット
  → 全APIリクエストに CF-Access-Authenticated-User-Email ヘッダー付与
  → Worker パススルー → Railway バックエンド
  → require_auth() でヘッダーからメール抽出
  → DB クエリが .eq("user_id", メール) でフィルタ
```

#### インフラ・DB修正

| 対応 | 詳細 |
|------|------|
| Railway 環境変数 | `ENVIRONMENT=production` を追加。未認証リクエストは 401 拒否 |
| Supabase データ移行 | `dev@localhost` → `ryu3ta.ke.mo100307@gmail.com` に UPDATE（holdings, trades, cash_balances, portfolio_snapshots） |
| Supabase スキーマ修正 | `user_settings`, `user_watchlists` の `user_id` カラムを UUID → TEXT に変換（外部キー制約を削除してから） |

#### レースコンディション対応

初回デプロイ後、保有タブで `401 Authentication required` エラーが発生。原因は SWR が UserProvider のメール取得完了前に API を叩いていたため。

- `layout.tsx` のプロバイダー順序を `SWRProvider > UserProvider` に変更
- UserProvider 内で `useSWRConfig().mutate(() => true)` を呼び、メール取得後に全 SWR キーを再検証

### 影響範囲

認証が必要な全エンドポイント（`require_auth` 使用箇所）:

- `/api/holdings/*` — 保有銘柄、現金残高、ポートフォリオ履歴
- `/api/trades/*` — 取引履歴、統計、売却
- `/api/watchlist/*` — ウォッチリスト CRUD

#### ヘッダー名変更: CF-Access-* → X-User-Email

上記の修正では `CF-Access-Authenticated-User-Email` ヘッダーを使用していたが、**Cloudflare エッジが `CF-*` プレフィックスのヘッダーを自動的にストリップ**するため、Worker 経由でバックエンドに到達しなかった。

- Worker 経由のリクエストはすべて 401 になる
- Railway 直接アクセス（curl）ではヘッダーが通る
- 原因: `CF-*` は Cloudflare 予約ヘッダー。偽装防止のためエッジで除去される

**対応**: ヘッダー名を `X-User-Email` に変更（frontend, worker, backend 全箇所）

#### レースコンディション対応（改善）

`mutate(() => true)` による事後的な再検証から、**UserProvider が `isLoading` 中は子コンポレンダリングをブロック**する構造的な解決に変更。

```
SWRProvider > UserProvider(isLoading中は子をブロック) > children
```

メール取得完了後に初めて子がレンダリングされるため、SWR フックは常に auth-store にメールがセットされた状態で発火する。

#### 認証フロー（最終版）

```
ブラウザ → Cloudflare Access ログイン
  → /cdn-cgi/access/get-identity でメール取得
  → auth-store にセット
  → UserProvider が isLoading 解除 → 子コンポーネントレンダリング開始
  → 全APIリクエストに X-User-Email ヘッダー付与
  → Worker パススルー → Railway バックエンド
  → require_auth() で X-User-Email ヘッダーからメール抽出
  → DB クエリが .eq("user_id", メール) でフィルタ
```

### 教訓

1. **Cloudflare Access はアクセスゲートであり、API認証ではない** — Access で許可されたユーザーでも、APIレベルでのユーザー識別は別途必要
2. **異なるドメイン間では Cloudflare Access のヘッダー自動付与が効かない** — Pages と Worker が別ドメインの場合、フロントエンドが明示的にヘッダーを送る必要がある
3. **開発用フォールバック (`dev@localhost`) は本番で無効にすべき** — `ENVIRONMENT=production` の設定漏れで全ユーザーが同一ユーザーとして扱われた
4. **SWR の非同期認証にはレースコンディション対策が必要** — プロバイダーの順序と再検証タイミングを適切に設計する
5. **`CF-*` ヘッダーは Cloudflare 予約済みで外部から送信不可** — カスタムヘッダーには `X-` プレフィックスを使用すること

## 2026-02-26: Railway バックエンドへの直接アクセスで全ユーザーデータ漏洩

### 問題

Railway バックエンドの URL に直接 `X-User-Email` ヘッダーを偽装した curl リクエストを送るだけで、**任意のユーザーの保有銘柄・取引・ウォッチリストのデータが取得できた**。

```
curl -H "X-User-Email: victim@example.com" \
  https://empathetic-hope-production.up.railway.app/api/holdings
→ 200 OK（保有銘柄21件の全データが返却）
```

### 原因

- Cloudflare Access はフロントエンド（Pages）のみを保護しており、バックエンド API は**完全にオープン**
- `require_auth()` は `X-User-Email` ヘッダーを信頼するだけで、リクエスト元の検証を行っていなかった
- CORS はブラウザのみの制約であり、サーバーサイド（curl 等）からの攻撃は防げない

```
攻撃者 → curl + 偽ヘッダー → Railway 直接 → auth.py（ヘッダー信頼）→ 全データ返却
```

### 対応

Worker ↔ Backend 間に **共有シークレット (`X-Proxy-Secret`)** を追加し、正規の Worker 経由以外のリクエストを拒否。

| ファイル | 変更内容 |
|----------|----------|
| `app/backend/auth.py` | `PROXY_SECRET` 環境変数を読み取り、`require_auth()` で `X-Proxy-Secret` ヘッダーを `hmac.compare_digest` で検証。不一致は 403 |
| `app/worker/src/index.ts` | `proxyHeaders` に `X-Proxy-Secret: env.PROXY_SECRET` を付与 |
| Railway 環境変数 | `PROXY_SECRET` を追加 |
| Cloudflare Worker Secret | `wrangler secret put PROXY_SECRET` で同じ値を設定 |

#### 追加修正

| ファイル | 変更内容 |
|----------|----------|
| `app/backend/routers/watchlist.py` | `add_ticker` / `remove_ticker` の update クエリに `.eq("user_id", user_email)` を追加（防御的修正） |

#### 認証フロー（最終版）

```
ブラウザ → Cloudflare Access ログイン
  → /cdn-cgi/access/get-identity でメール取得
  → auth-store にセット
  → UserProvider が isLoading 解除 → 子コンポーネントレンダリング開始
  → 全APIリクエストに X-User-Email ヘッダー付与
  → Worker が X-Proxy-Secret ヘッダーを付与 → Railway バックエンド
  → require_auth() で:
    1. X-Proxy-Secret を PROXY_SECRET と照合（不一致→403）
    2. X-User-Email からメール抽出
  → DB クエリが .eq("user_id", メール) でフィルタ
```

#### 攻撃テスト結果（修正後）

| テスト | 結果 |
|--------|------|
| Railway 直接 + 偽メール（シークレットなし）| **403 Forbidden** |
| Railway 直接 + 偽メール + 偽シークレット | **403 Forbidden** |
| Worker 経由 + 正規メール | **200 OK**（正常動作） |
| SQLインジェクション（ticker: `AAPL' OR 1=1--`）| **400 Bad Request**（バリデーション拒否） |

### 教訓

1. **バックエンド URL が知られたら終わり** — Cloudflare Access はフロントのみ保護。バックエンドの API は別途保護が必要
2. **CORS はサーバーサイド攻撃を防げない** — curl、Postman、スクリプトからのリクエストには CORS は無力
3. **Worker ↔ Backend 間は共有シークレットで信頼関係を構築** — `X-Proxy-Secret` ヘッダーでリクエスト元を検証
4. **タイミング攻撃対策** — シークレット比較には `hmac.compare_digest` を使用（定数時間比較）
