# システム設計 詳細 2 — CRUD Worker 移行 & Cloud Run 移行

> 作成日: 2026-03-01
> 対象: Step 1（CRUD → Cloudflare Workers）+ Step 2（Railway → Google Cloud Run）

---

## 1. 移行の全体像

### 1.1 なぜ移行したか

| 問題 | 影響 | 解決策 |
|------|------|--------|
| Railway が全 84 エンドポイントを処理 | CRUD（SELECT するだけ）に Python コンテナ 200MB 常駐 | CRUD 49 個を Worker に移行 |
| Railway は無料トライアル限定（$5 クレジット） | トライアル終了後は課金必須 | Cloud Run（200 万 req/月 無料）に移行 |
| CRUD が 3 ホップ（Browser→Worker→Railway→Supabase） | レイテンシ ~400ms | Worker→Supabase 直接で ~100ms |
| インメモリキャッシュが Railway 再起動で消失 | ユーザー体験の劣化 | Cloud Run + 将来的に Upstash Redis |

### 1.2 移行前後の比較

```
【移行前】全リクエストが Railway 経由
  Browser → CF Worker → Railway (Python 84ep) → Supabase

【移行後】CRUD と計算を分離
  Browser → CF Worker ─┬─ CRUD 49ep → Supabase 直接（Worker 内処理）
                       └─ 計算 35ep → Cloud Run (Python) → yfinance/Supabase
```

### 1.3 コスト変化

| サービス | 移行前 | 移行後 |
|---------|--------|--------|
| Railway | $5/月〜 | $0（廃止） |
| Cloud Run | — | $0（無料枠: 200 万 req/月） |
| CF Workers | $0 | $0 |
| CF Pages | $0 | $0 |
| Supabase | $0 | $0 |
| **合計** | **$5/月〜** | **$0** |

CI（`--update-env-vars`）で設定:

| 変数 | 設定方法 | 用途 |
|------|----------|------|
| `ENVIRONMENT` | CI で自動設定 | `production` |
| `UPSTASH_REDIS_REST_URL` | CI (GitHub Secrets 経由) | Upstash Redis REST URL |
| `UPSTASH_REDIS_REST_TOKEN` | CI (GitHub Secrets 経由) | Upstash Redis REST トークン |

手動で設定（`gcloud run services update --update-env-vars`）:

| 変数 | 用途 |
|------|------|
| `PROXY_SECRET` | Worker との共有シークレット（CSRF 対策） |
| `SUPABASE_URL` | Supabase プロジェクト URL |
| `SUPABASE_KEY` | Supabase service_role key |
| `SUPABASE_JWT_SECRET` | JWT 署名検証用シークレット |
| `ADMIN_EMAILS` | 管理者メールアドレス（カンマ区切り） |
| `MFA_ENCRYPTION_KEY` | MFA TOTP 暗号化キー (hex 64 文字) |

**重要**: CI の `--update-env-vars` は指定した変数のみ更新し、手動設定した変数は保持される。`--set-env-vars` だと全変数が上書きされるので使わないこと。

### 3.6 Dockerfile

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8080
CMD ["sh", "-c", "uvicorn main:app --host 0.0.0.0 --port ${PORT:-8080}"]
```

- Cloud Run は `PORT` 環境変数を自動注入（デフォルト 8080）
- `${PORT:-8080}` でローカル開発時も 8080 で起動

### 3.7 起動時検証

`main.py` の `lifespan()` で本番環境（`ENVIRONMENT=production`）の場合:
- `PROXY_SECRET` が未設定 → `RuntimeError` で起動失敗
- `SUPABASE_JWT_SECRET` が未設定 → `RuntimeError` で起動失敗

これにより、シークレット未設定のまま本番稼働することを防止。

## 4. Workload Identity Federation (WIF) — 認証設定

### 4.1 WIF とは

GitHub Actions から GCP に**キーレス認証**する仕組み。
サービスアカウントキー（JSON ファイル）を GitHub Secrets に保存する必要がない。

```
GitHub Actions
  ↓ OIDC トークン発行
GitHub OIDC Provider
  ↓ トークン検証
GCP Workload Identity Pool
  ↓ 条件一致（リポジトリ名）
GCP Service Account
  ↓ 権限付与
Cloud Run / Artifact Registry
```

### 4.2 WIF 構成要素

| 要素 | 名前 | 説明 |
|------|------|------|
| Workload Identity Pool | `github` | GitHub からの認証リクエストを受け付けるプール |
| OIDC Provider | `github-actions` | GitHub Actions の OIDC トークンを検証 |
| Service Account | `github-actions-deployer` | デプロイ操作を実行する GCP アカウント |

### 4.3 OIDC Provider 設定

| 設定 | 値 |
|------|-----|
| Issuer URI | `https://token.actions.githubusercontent.com` |
| Attribute Mapping | `google.subject=assertion.sub, attribute.repository=assertion.repository` |
| Attribute Condition | `assertion.repository=='Urban-Sea/-'` |

`attribute.repository` でリポジトリを制限し、他のリポジトリからのデプロイを防止。

### 4.4 Service Account ロール

| ロール | 用途 |
|--------|------|
| `roles/run.admin` | Cloud Run サービスの作成・更新・削除 |
| `roles/artifactregistry.writer` | Docker イメージのプッシュ |
| `roles/iam.serviceAccountUser` | サービスアカウントとしてデプロイ実行 |
| `roles/iam.workloadIdentityUser` | WIF 経由でのトークン取得 |

### 4.5 WIF ↔ SA バインディング

```
principalSet://iam.googleapis.com/projects/1073412395842/locations/global/workloadIdentityPools/github/attribute.repository/Urban-Sea/-
```

このバインディングにより、`Urban-Sea/-` リポジトリの GitHub Actions のみが `github-actions-deployer` サービスアカウントの権限を使用できる。

### 4.6 GitHub Secrets

| Secret 名 | 値 |
|-----------|-----|
| `GCP_PROJECT_ID` | `open-regime` |
| `GCP_WORKLOAD_IDENTITY_PROVIDER` | `projects/1073412395842/locations/global/workloadIdentityPools/github/providers/github-actions` |
| `GCP_SERVICE_ACCOUNT` | `github-actions-deployer@open-regime.iam.gserviceaccount.com` |

## 5. CI/CD パイプライン

### 5.1 deploy.yml の構成

```
on:
  push:
    branches: [main]
    paths:
      - 'app/worker/**'
      - 'app/frontend/**'
      - 'app/admin-frontend/**'
      - 'app/backend/**'

jobs:
  changes:        ← パス変更検知
  deploy-worker:  ← Worker デプロイ (npx wrangler deploy)
  deploy-pages:   ← Frontend デプロイ (wrangler pages deploy)
  deploy-admin:   ← Admin デプロイ (wrangler pages deploy)
  deploy-backend: ← Cloud Run デプロイ (gcloud run deploy)  ★新規
```

### 5.2 deploy-backend ジョブの流れ

```
1. actions/checkout@v4
   ↓
2. google-github-actions/auth@v2
   WIF でキーレス認証
   ↓
3. google-github-actions/setup-gcloud@v2
   gcloud CLI セットアップ
   ↓
4. gcloud auth configure-docker
   Artifact Registry への Docker 認証
   ↓
5. docker build & push
   app/backend/ をビルド
   タグ: :<commit-sha> + :latest
   → us-east1-docker.pkg.dev/open-regime/open-regime/backend
   ↓
6. gcloud run deploy
   Cloud Run にデプロイ
   --update-env-vars ENVIRONMENT=production
   ↓
7. gcloud run services describe
   デプロイ先 URL を表示
```

### 5.3 トリガー条件

| 変更パス | 実行されるジョブ |
|---------|----------------|
| `app/worker/**` | deploy-worker |
| `app/frontend/**` | deploy-pages |
| `app/admin-frontend/**` | deploy-admin |

## 6. リクエストフロー詳細

### 6.1 CRUD リクエスト（例: GET /api/holdings）

```
Browser
  │ Authorization: Bearer <JWT>
  ▼
CF Worker (isComputePath=false, CRUD_IN_WORKER=true)
  │ 1. CORS 検証
  │ 2. JWT 検証（Worker 内）
  │ 3. Supabase に直接 SELECT
  │ 4. レスポンス返却
  ▼
Browser (レイテンシ: ~100ms)
```

Cloud Run には到達しない。

### 6.2 計算リクエスト（例: GET /api/signal/NVDA）

```
Browser
  │ Authorization: Bearer <JWT>
  ▼
CF Worker (isComputePath=true)
  │ 1. CORS 検証
  │ 2. Cache API チェック（signal は TTL=0 なので MISS）
  │ 3. Cloud Run にプロキシ
  │    Headers: Authorization, X-Proxy-Secret, X-MFA-Token
  ▼
Cloud Run (Python/FastAPI)
  │ 1. CSRFOriginMiddleware: X-Proxy-Secret HMAC 検証
  │ 2. auth.py: JWT 検証 → ユーザー解決
  │ 3. redis_cache: L1 (インメモリ) → L2 (Redis) チェック
  │    → HIT: キャッシュから即座に返却 (~100ms)
  │    → MISS: yfinance.download() → pandas 計算 → Redis に保存
  │ 4. レスポンス返却
  ▼
CF Worker
  │ セキュリティヘッダー付与
  │ X-Cache: MISS
  ▼
Browser (レイテンシ: ~1-3 秒)
```

### 6.3 キャッシュ済みリクエスト（例: GET /api/liquidity/overview）

```
Browser
  │ Authorization: Bearer <JWT>
  ▼
CF Worker
  │ 1. CORS 検証
  │ 2. Cache API チェック → HIT (TTL: 24 時間)
  │ 3. キャッシュからレスポンス返却
  │    X-Cache: HIT
  ▼
Browser (レイテンシ: ~60ms)
```

Cloud Run には到達しない。

## 7. セキュリティ機構

### 7.1 認証フロー

```
【Worker 内 CRUD】
  JWT → Worker で検証 → Supabase 直接クエリ

【Cloud Run 計算】
  JWT → Worker が転送 → Cloud Run で再検証
  X-Proxy-Secret → Worker が付与 → Cloud Run で HMAC 検証
```

### 7.2 X-Proxy-Secret の仕組み

Worker と Cloud Run が同じシークレットを共有。

1. Worker: リクエスト転送時に `X-Proxy-Secret` ヘッダーを付与（[proxy.ts:55](../../app/worker/src/routes/proxy.ts#L55)）
2. Cloud Run: `CSRFOriginMiddleware` で `hmac.compare_digest()` による定数時間比較（[main.py:59-64](../../app/backend/main.py#L59-L64)）
3. 不一致 → 403 Forbidden

これにより Cloud Run の URL を直接叩いても、Worker を経由しないリクエストは拒否される。

### 7.3 廃止したもの

| 機能 | 理由 |
|------|------|
| X-User-Email ヘッダー認証 | Cloudflare Access 時代のレガシー。JWT 認証に統一 |
| auth.py の `_resolve_user_id()` | X-User-Email → UUID 解決。不要になった |
| auth.py の `_user_cache` | email→UUID キャッシュ。不要になった |
| railway.json | Railway 固有の設定ファイル |

### 7.4 現在のセキュリティ対策一覧

| 対策 | 実装場所 | 説明 |
|------|----------|------|
| CORS | Worker (cors.ts) | ALLOWED_ORIGIN 以外は null 返却 |
| CSRF | Cloud Run (main.py) | X-Proxy-Secret HMAC 検証 |
| JWT 検証 | Worker (CRUD) + Cloud Run (計算) | Supabase ES256 JWKS / HS256 |
| レート制限 | Worker (120/min, Redis INCR) + Cloud Run (60/min) | IP ベース |
| セキュリティヘッダー | Worker + Cloud Run 両方 | nosniff, DENY, HSTS 等 |
| 起動時検証 | Cloud Run (main.py lifespan) | 必須シークレット未設定で起動拒否 |
| email 確認チェック | Cloud Run (auth.py) | email_confirmed_at 必須 |
| アルゴリズム混乱防止 | Cloud Run (auth.py) | JWT alg に応じて JWKS or HMAC を選択 |

│  ─────────────────────────────────────────────────       │
│  1. UserProvider: /cdn-cgi/access/get-identity → email   │
│  2. auth-store に email を保存                            │

                           ▼
┌─────────────────────────────────────────────────────────┐
│  Layer 3: TOTP MFA（Worker 内）                           │
│  ─────────────────────────────────────────────────       │
│  MfaGate コンポーネントが制御:                            │
│                                                          │
│  [初回セットアップ]                                       │

### 9.4 MFA セキュリティ詳細

| 項目 | 実装 |
|------|------|
| TOTP アルゴリズム | RFC 6238 (6 桁, 30 秒間隔) |
| シークレット保存 | AES 暗号化（MFA_ENCRYPTION_KEY）で admin_mfa テーブルに保存 |
| セッショントークン | 64 文字 hex（256 bit ランダム）、SHA-256 ハッシュで DB に保存 |
| セッション有効期間 | 1 時間 |
| ブルートフォース保護 | 15 分間で最大 5 回の試行（超過 → 429 Too Many Requests） |
| リプレイ防止 | 使用済みコードを 90 秒間記録し再使用を拒否 |
| ログアウト | サーバー側でセッション削除 + CF Access ログアウト |

### 9.5 監査ログ

```

  target_type   — "user", "feature_flag"
  target_id     — 対象の ID
  old_value     — 変更前の値 (JSON)
  new_value     — 変更後の値 (JSON)
  created_at    — タイムスタンプ
```

## 10. Resend SMTP（Step 4.5）

### 10.1 概要

Supabase Auth はデフォルトで内蔵 SMTP を使用するが、**無料プランでは月 4 通の送信制限**がある。
Resend の Custom SMTP に差し替えることで **月 3,000 通（100 通/日）** に拡大。

- コード変更なし、CI/CD 変更なし、デプロイ不要
- Supabase ダッシュボードで SMTP サーバーを Resend に切り替えるだけ

### 10.2 構成

```
Supabase Auth
  │ サインアップ確認、パスワードリセット等のメール送信時
  ▼
Resend SMTP (smtp.resend.com:465)
  │ Resend がメール配信を処理
  ▼
ユーザーのメールボックス
  送信元: noreply@open-regime.com (Open Regime)
```

### 10.3 Supabase SMTP 設定値

| 設定項目 | 値 |
|---------|-----|
| Sender email | `noreply@open-regime.com` |
| Sender name | `Open Regime` |
| Host | `smtp.resend.com` |
| Port | `465` |
| Username | `resend` |
| Password | Resend API Key（`re_xxxxxxxx`） |

### 10.4 カスタムドメイン + メール認証

| DNS レコード | 種類 | 用途 |
|-------------|------|------|
| `resend._domainkey.open-regime.com` | TXT | DKIM 署名 |
| `send.open-regime.com` | TXT | SPF (amazonses.com) |
| `send.open-regime.com` | MX | バウンス処理 |
| `_dmarc.open-regime.com` | TXT | DMARC ポリシー |

### 10.5 Resend 無料枠

| 項目 | 制限 |
|------|------|
| 送信数 | 3,000 通/月（100 通/日） |
| ドメイン | 1 つ（`open-regime.com`） |

### 10.6 送信されるメールの種類

| メール種別 | トリガー |
|-----------|---------|
| サインアップ確認 | 新規ユーザー登録時 |
| パスワードリセット | 「パスワードを忘れた」操作時 |
| メールアドレス変更確認 | メールアドレス変更時 |
| パスワード変更通知 | パスワード変更時 |
| メールアドレス変更通知 | メールアドレス変更時 |
