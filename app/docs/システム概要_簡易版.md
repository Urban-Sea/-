## 2. レスポンス時間の内訳（現状）

### 体感で遅いエンドポイント

| エンドポイント | 合計時間 | 内訳 |
|--------------|---------|------|
| `/api/signal/{ticker}` | **600-900ms** | RTT 150ms + yfinance 200ms + BOS/CHoCH計算 200ms + キャッシュミス時 |
| `/api/signal/batch` (20銘柄) | **3-5s** | 20× yfinance + 20× シグナル計算（順次実行） |
| `/api/signal/{ticker}/history` | **1-2s** | yfinance 2年分取得 + SPYデータ + 全日走査 |
| `/api/liquidity/plumbing-summary` | **500-800ms** | RTT 150ms + 11 DB並列クエリ + Layer計算 |
| `/api/employment/risk-score` | **400-700ms** | RTT 150ms + 5 DB並列クエリ + スコア計算 |
| `/api/stock/{ticker}/history` | **300-500ms** | RTT 150ms + yfinance履歴取得 |

### 速いエンドポイント（キャッシュヒット時）

| エンドポイント | 合計時間 | 理由 |
|--------------|---------|------|
| `/api/regime` | **200ms** | RTT + キャッシュヒット |
| `/api/stocks` | **200ms** | DB直クエリ（軽量） |
| `/api/employment/overview` | **200ms** | DB直クエリ（軽量） |

### レイテンシの最大要因

```
1位: ネットワーク RTT（日本→US East）     150-200ms  ← 物理的に避けられない
2位: yfinance 外部API呼び出し             50-150ms/call
3位: CPU計算（BOS/CHoCH/V10シグナル）     100-300ms
4位: DB並列クエリ（Supabase）             50-200ms
```

## 3. キャッシュ戦略（現状 — 2026-02-25更新）

| レイヤー | 場所 | TTL | 対象 |
|---------|------|-----|------|
| **L1: ブラウザ (SWR)** | クライアント | **セッション中永続** ✅ | 全APIレスポンス（15フック） |
| **L2: Edge** | Cloudflare | なし | なし（Pages = 静的のみ） |
| **L3: バックエンド** | Railway メモリ | 5分〜1時間 | signal, stock, liquidity, employment |
| **L4: DB事前計算** | Supabase | **24時間** ✅ | risk-score, plumbing-summary, events, policy-regime |

**改善済み (2026-02-25)**:
- L1: SWR導入 → 2回目以降のページ表示 **0ms**（キャッシュヒット即表示）
- L4: バッチ事前計算 → 重い4エンドポイントがDB直読み **~50ms**
- タブ切替・ページ戻りが瞬時

## 4. パフォーマンス改善（実施済み — 2026-02-25）

### Phase 1: SWR導入 ✅

`useSWR` を全ページに導入（15フック）。`useEffect + fetch` を置き換え。

```
改善結果:
- 2回目以降のページ表示: 600-900ms → 0ms（キャッシュヒット即表示）
- タブ切替: 500ms → 0ms
- バックグラウンド再検証でデータ鮮度も維持
```

**SWRフック一覧** (`app/frontend/src/lib/api.ts`):
`useRegime`, `useLatestMarketState`, `useLiquidityOverview`, `usePlumbingSummary`,
`useMarketEvents`, `usePolicyRegime`, `useHistoryCharts`, `useBacktestStates`,
`useEmploymentRiskScore`, `useRiskHistory`, `useHoldings`, `useTrades`,
`useTradeStats`, `useStocks`, `useBatchQuotes`

**SWR設定** (`app/frontend/src/lib/swr.tsx`):
- `revalidateOnFocus: false` — フォーカス時の自動再取得を無効
- `dedupingInterval: 2000ms` — 重複リクエスト防止
- `errorRetryCount: 2` — エラー時リトライ回数

### Phase 2: バッチ事前計算 ✅

重い4エンドポイントの計算結果をSupabaseに事前保存。APIはDB直読みのみで応答。

```
改善結果:
- risk-score:       500ms → ~50ms（DB直読み）
- plumbing-summary: 800ms → ~50ms
- events:           400ms → ~50ms
- policy-regime:    500ms → ~50ms
```

**仕組み:**
1. `python app/batch/run.py --precompute` がバックエンドAPIを呼び出し
2. レスポンスJSONを `precomputed_results` テーブルにupsert
3. `--daily` / `--weekly` 実行時に自動で事前計算
4. バックエンドは事前計算結果があれば直読み（24時間TTL）、なければフォールバック計算

### Phase 3: Cloudflare Workers（将来検討）

SWR + 事前計算で十分高速。Workers導入時の改善幅はRTT削減 150ms→10ms のみ。
個人利用では体感差が小さいため、現時点では不要。

## 5. セキュリティ（2026-02-25更新）

### 認証フロー
```
ユーザー → Cloudflare Access (メールOTP/Google OAuth)
         → Next.js (CF-Access-Authenticated-User-Email ヘッダー付与)
         → FastAPI (require_auth でヘッダー検証)
         → Supabase (service_role でRLSバイパス)
```

### Supabase キー使い分け
| 用途 | キー | RLS |
|------|------|-----|
| **バックエンド (Railway)** | `SUPABASE_KEY` (service_role) | バイパス（全操作可） |
| **バッチ (ローカル)** | `SUPABASE_KEY` (service_role) | バイパス（全操作可） |
| **直接PostgRESTアクセス** | anon key | RLS適用（SELECTのみ） |

### RLSポリシー（全20テーブル）
| テーブル群 | RLS | anonポリシー |
|-----------|-----|-------------|
| 流動性データ (8テーブル) | 有効 | `anon_select` (SELECT) |
| 雇用データ (3テーブル) | 有効 | `anon_select` (SELECT) |
| 履歴 (2テーブル) | 有効 | `anon_select` (SELECT) |
| 個人データ: holdings, trades, user_settings, user_watchlists | 有効 | `anon_select` (SELECT) |
| その他: stock_master, stock_cache, manual_inputs, precomputed_results, data_revisions | 有効 | `anon_select` (SELECT) |
