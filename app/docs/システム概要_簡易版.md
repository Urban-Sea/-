## 2. レスポンス時間の内訳（現状）

### 体感で遅いエンドポイント

| エンドポイント | 合計時間 | 内訳 |
|--------------|---------|------|
| `/api/signal/{ticker}` | **600-900ms** | RTT 150ms + yfinance 200ms + BOS/CHoCH計算 200ms + キャッシュミス時 |
| `/api/signal/batch` (20銘柄) | **3-5s** | 20× yfinance + 20× シグナル計算（順次実行） |
| `/api/signal/{ticker}/history` | **1-2s** | yfinance 2年分取得 + SPYデータ + 全日走査 |
| `/api/liquidity/plumbing-summary` | **500-800ms** | RTT 150ms + 11 DB並列クエリ + Layer計算 |
| `/api/employment/risk-score` | **400-700ms** | RTT 150ms + 5 DB並列クエリ + スコア計算 |
| `/api/stock/{ticker}/history` | **300-500ms** | RTT 150ms + yfinance履歴取得 |

### 速いエンドポイント（キャッシュヒット時）

| エンドポイント | 合計時間 | 理由 |
|--------------|---------|------|
| `/api/regime` | **200ms** | RTT + キャッシュヒット |
| `/api/stocks` | **200ms** | DB直クエリ（軽量） |
| `/api/employment/overview` | **200ms** | DB直クエリ（軽量） |

### レイテンシの最大要因

```
1位: ネットワーク RTT（日本→US East）     150-200ms  ← 物理的に避けられない
2位: yfinance 外部API呼び出し             50-150ms/call
3位: CPU計算（BOS/CHoCH/V10シグナル）     100-300ms
4位: DB並列クエリ（Supabase）             50-200ms
```

## 3. キャッシュ戦略（現状）

| レイヤー | 場所 | TTL | 対象 |
|---------|------|-----|------|
| **L1: ブラウザ** | クライアント | **なし** ← 問題 | なし |
| **L2: Edge** | Cloudflare | **なし** ← 問題 | なし（Pages = 静的のみ） |
| **L3: バックエンド** | Railway メモリ | 5分〜1時間 | signal, stock, liquidity, employment |
| **L4: DB** | Supabase | なし | バッチ結果のみ |

**問題点**: L1（ブラウザ）とL2（Edge）にキャッシュが一切ない。
同じページを再訪問するたびに毎回 日本→US East のフルラウンドトリップが発生。

## 4. パフォーマンス改善オプション

### Option A: フロントエンドにSWR/React Queryを追加（推奨）

```
効果: ★★★★★（最大の改善、最小のコスト）
工数: 2-3時間
```

**やること**: `fetch()` を `useSWR()` に置き換えるだけ
```
Before: ページ遷移のたびに → fetch → 150ms RTT + API処理
After:  初回だけfetch → 以降はブラウザキャッシュから即表示 + バックグラウンド再検証
```

**効果**:
- 2回目以降のページ表示: **0ms**（キャッシュヒット → 即表示）
- バックグラウンドで最新データを取得し、差分があれば更新
- タブ切替・ページ戻りが瞬時

### Option B: Cloudflare Workers（Edgeキャッシュプロキシ）

```
効果: ★★★☆☆（初回アクセスのRTT削減）
工数: 半日〜1日
```

**やること**: Workers で `/api/*` をプロキシし、レスポンスをEdgeにキャッシュ
```
Before: ブラウザ → Cloudflare Pages → Railway(US East) → 150ms RTT
After:  ブラウザ → Cloudflare Worker(Tokyo Edge) → キャッシュヒット → 10ms
                                                  → キャッシュミス → Railway → 150ms
```

**問題点**:
- バックエンド既にインメモリキャッシュあり（二重キャッシュ）
- Workers無料枠: 10万req/日（個人利用なら十分）
- キャッシュ無効化の管理が複雑になる
- yfinanceの計算結果はTTL 5分 → Edgeキャッシュと整合性の問題

### Option C: バックエンドの最適化（既に実施済み部分あり）

```
効果: ★★★☆☆
工数: 3-5時間
```

**やること**:
- signal/batch を並列実行に変更（現在は順次）
- yfinance結果をSupabase stock_cacheに保存（TTL 5分）
- 重い計算結果（risk-score, plumbing-summary）のTTLを30分→2時間に延長

### Option D: Railway リージョン変更（Tokyo）

```
効果: ★★☆☆☆（RTTのみ改善）
工数: 30分
```

**やること**: Railwayのリージョンを US East → Asia (Tokyo) に変更
```
Before: 日本→US East RTT ≈ 150-200ms
After:  日本→Tokyo RTT ≈ 10-20ms
```

**問題点**:
- Supabaseが US East のまま → Railway(Tokyo)→Supabase(US East) で逆にDB遅延増加
- yfinance APIは米国サーバー → 変わらない
- 全体的にはむしろ遅くなる可能性あり

## 5. 推奨プラン

### Phase 1: SWR導入（即効性、最大効果）← まずこれ

`app/frontend/src/lib/api.ts` に SWR ラッパーを追加。
全ページの `useEffect + fetch` を `useSWR` に置き換え。

```
期待される改善:
- 2回目以降のページ表示: 600-900ms → 0ms（即時表示）
- タブ切替: 500ms → 0ms
- バックグラウンド再検証でデータ鮮度も維持
```

### Phase 2: バッチ並列化 + キャッシュTTL延長（補助）

- `/api/signal/batch` を asyncio.gather で並列実行
- liquidity/employment の TTL を延長（データ更新頻度に合わせる）

### Phase 3: Cloudflare Workers（必要に応じて）

SWR導入後でもまだ初回表示が遅い場合のみ検討。
個人利用なら SWR だけで十分快適になる見込み。

---

## 6. 結論: Cloudflare Workers は今は不要

| 比較 | SWR (推奨) | Workers |
|------|-----------|---------|
| 2回目以降の速度 | **0ms** | 10-50ms |
| 初回アクセス | 変わらない | 10-50ms改善 |
| 実装コスト | 2-3時間 | 半日〜1日 |
| 保守コスト | なし | キャッシュ管理 |
| 効果の大きさ | **★★★★★** | ★★★☆☆ |

**理由**:
1. 最大のボトルネックは「同じデータを毎回fetchすること」→ SWRで解決
2. Workers は初回アクセスのRTTしか改善しない（150ms→10ms）
3. バックエンドに既にインメモリキャッシュがある → Edgeキャッシュは二重管理
4. 個人利用のアプリ → 同じユーザーが何度も同じページを見る → SWR の効果が最大
