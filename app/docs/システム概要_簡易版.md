## 2. レスポンス時間（Worker導入後 — 2026-02-25更新）

### キャッシュ対象エンドポイント（Worker HIT時）

| エンドポイント | Worker (cached) | 直接 Railway | 改善 |
|--|--|--|--|
| `/api/regime` | **64ms** | 975ms | **15倍** |
| `/api/employment/risk-score` | **69ms** | 2,651ms | **38倍** |
| `/api/liquidity/plumbing-summary` | **59ms** | 1,477ms | **25倍** |
| `/api/employment/risk-history` | **53ms** | 1,036ms | **20倍** |

### キャッシュ非対象（パススルー）

| エンドポイント | 時間 | 理由 |
|--|--|--|
| `/api/signal/{ticker}` | 600-900ms | リアルタイム計算（yfinance + V10） |
| `/api/holdings/*` | 300-500ms | ユーザーCRUD |
| `/api/stock/{ticker}/history` | 300-500ms | yfinance都度取得 |

### レイテンシの最大要因

```
キャッシュMISS時:
  1位: ネットワーク RTT（CF backbone → US East） ~300ms
  2位: yfinance 外部API呼び出し                  50-150ms/call
  3位: CPU計算（BOS/CHoCH/V10シグナル）           100-300ms

キャッシュHIT時:
  全て ~60ms（東京PoP NRT から即返却）
```

## 3. キャッシュ戦略（4層 — 2026-02-25更新）

| レイヤー | 場所 | TTL | 対象 |
|---------|------|-----|------|
| **L1: ブラウザ (SWR)** | クライアント | **セッション中永続** ✅ | 全APIレスポンス（15フック） |
| **L2: Edge (Worker)** | Cloudflare 東京PoP | **5分〜4時間** ✅ | 日次/週次データ（12エンドポイント） |
| **L3: バックエンド** | Railway メモリ | 5分〜1時間 | signal, stock |
| **L4: DB事前計算** | Supabase | **24時間** ✅ | risk-score, plumbing-summary, events, policy-regime |

**Worker TTL一覧:**

| TTL | エンドポイント |
|-----|--------------|
| 4時間 | risk-score, plumbing-summary, events, policy-regime |
| 2時間 | risk-history, backtest-states |
| 1時間 | history-charts, stocks |
| 30分 | employment/overview |
| 15分 | liquidity/overview |
| 5分 | regime, market-state/latest |
| なし | holdings/*, trades/*, signal/*, stock/*, exit/* |

## 5. CI/CD & 自動化（2026-02-25）

### デプロイ（git push トリガー）
```
git push to main
  ├── Railway: バックエンド自動デプロイ（Railway Git連携）
  └── GitHub Actions (.github/workflows/deploy.yml)
       ├── app/worker/** 変更 → Worker 自動デプロイ (~25秒)
       └── app/frontend/** 変更 → Pages 自動ビルド＆デプロイ
```

### 日次バッチ（cron 自動実行）
```
毎朝 7:30 JST (.github/workflows/batch-daily.yml)
  └── GitHub Actions cron (schedule: 22:30 UTC)
       ├── FRED/Yahoo/NY Fed データ取得 → Supabase upsert
       ├── precompute_all() → 重い計算結果をDB保存
       └── _warmup_cache() → Worker全エンドポイントのキャッシュ充填
           → 朝一番のユーザーも ~60ms で高速表示
```

※ `workflow_dispatch` 対応: GitHub UIから手動実行も可能

**必要な GitHub Secrets:**
- `CLOUDFLARE_API_TOKEN` (Workers Scripts: Edit, Pages: Edit, Account Settings: Read)
- `CLOUDFLARE_ACCOUNT_ID` (`c1a8026d00eb674a5d2750d6ef1527ba`)
- `SUPABASE_URL` (Supabase プロジェクトURL)
- `SUPABASE_KEY` (service_role キー)
- `FRED_API_KEY` (FRED APIキー)

## 6. セキュリティ

### 認証フロー
```
ユーザー → Cloudflare Access (メールOTP/Google OAuth)
         → Next.js (CF-Access-Authenticated-User-Email ヘッダー付与)
         → Worker (CORSヘッダー付与、キャッシュ管理)
         → FastAPI (require_auth でヘッダー検証)
         → Supabase (service_role でRLSバイパス)
```

### Supabase キー使い分け
| 用途 | キー | RLS |
|------|------|-----|
| **バックエンド (Railway)** | `SUPABASE_KEY` (service_role) | バイパス（全操作可） |
| **バッチ (ローカル)** | `SUPABASE_KEY` (service_role) | バイパス（全操作可） |
| **直接PostgRESTアクセス** | anon key | RLS適用（SELECTのみ） |
