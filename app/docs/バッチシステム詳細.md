# バッチデータパイプライン 詳細設計

## 概要

`app/batch/` は外部データソース（FRED、Yahoo Finance、NY Fed）から金融データを取得し、
Supabase テーブルに upsert した後、Layer Stress / Market State を月次計算するバッチパイプライン。

upsert 時に既存データと比較し、**上方修正/下方修正を自動検知**して `data_revisions` テーブルに記録する。

```
外部API                    Supabase テーブル              派生テーブル
─────────                  ──────────────                ──────────
FRED API ───→ fetchers/ ──→ fed_balance_sheet            layer_stress_history
              fred.py       interest_rates     ─→ calculators/ ─→
                            credit_spreads        layer_stress.py  market_state_history
                            mmf_assets
                            weekly_claims      ─→ 修正検知 ─→ data_revisions
                            economic_indicators

Yahoo    ───→ fetchers/ ──→ market_indicators
              yahoo.py      bank_sector

NY Fed   ───→ fetchers/ ──→ srf_usage
              nyfed.py

手動     ───→ POST API ──→ margin_debt          (FINRA: 公開APIなし)
                            economic_indicators  (ADP/Challenger: FRED遅延 or 系列なし)
```

## CLI 使い方

```bash
# 実行
python app/batch/run.py                     # 全実行（3年分 + Layer再計算）
python app/batch/run.py --daily             # 日次: Yahoo(14日) + FRED/SRF(3年分) + 週次失業保険
python app/batch/run.py --weekly            # 週次: FRBバランスシート, SRF, MMF, 雇用統計 + Layer再計算
python app/batch/run.py --fred              # FRED のみ（流動性4テーブル）
python app/batch/run.py --yahoo             # Yahoo のみ
python app/batch/run.py --srf              # SRF のみ
python app/batch/run.py --employment       # 雇用データのみ（weekly_claims + economic_indicators）
python app/batch/run.py --calc             # Layer計算のみ
python app/batch/run.py --full             # 15年分フル取得 + Layer計算
python app/batch/run.py --since 2025-01-01 # 指定日以降
python app/batch/run.py --verbose          # DEBUG ログ出力
```

| フラグ | 取得範囲 | 対象データ | Layer計算 |
|--------|----------|-----------|-----------|
| (なし) | 3年分 | 全テーブル（流動性+雇用） | する |
| `--daily` | FRED 3年/Yahoo 14日 | 金利, スプレッド, SRF, 失業保険, 市場指標 | しない |
| `--weekly` | 3年分 | FRBバランスシート, SRF, MMF, 雇用統計 | する |
| `--fred` | 3年分 | FRED流動性系列（4テーブル） | しない |
| `--yahoo` | 3年分 | Yahoo全ティッカー | しない |
| `--srf` | 3年分 | SRF のみ | しない |
| `--employment` | 3年分 | weekly_claims + economic_indicators | しない |
| `--calc` | 取得なし | — | する（2010年〜） |
| `--full` | 15年分 | 全テーブル | する |
| `--since YYYY-MM-DD` | 指定日〜 | 全テーブル | しない |

### 手動データ管理CLI (`manual_input.py`)

乖離スコア計算に必要な手動データ（ADP、Challenger、Truflation）を `manual_inputs` テーブルに投入するCLIツール。

```bash
# データ一覧表示
python app/batch/manual_input.py list

# 個別データ追加
python app/batch/manual_input.py add ADP 2026-01 122           # ADP月次変化（千人）
python app/batch/manual_input.py add CHALLENGER 2026-01 38792  # Challenger解雇件数
python app/batch/manual_input.py add TRUFLATION 2026-01 3.2    # Truflationインフレ率（%）

# ADP過去データ一括投入（CSVから水準→月次変化に自動変換）
python app/batch/manual_input.py load-adp
```

#### 手動データがリスクスコアに与える影響

| metric | 乖離スコア | 計算方法 | 影響 |
|--------|-----------|---------|------|
| ADP_CHANGE | 雇用乖離 (5点) | NFP 3M平均 vs ADP 3M平均の差をシグモイド変換 | 公式NFPの信頼度を測定 |
| CHALLENGER_CUTS | 雇用乖離 (5点) | 50K件超 & NFP>0 → 矛盾として加算 | 大量レイオフなのに雇用増=怪しい |
| TRUFLATION | インフレ乖離 (5点) | Truflation - コアCPI YoY → 線形スケール | 公式CPIが実感より低いかを検知 |

> **手動データがないと乖離スコアは0点になる。** 自動取得されるICSA（失業保険）のみで部分動作可能。

## スケジューリング

### 推奨 cron 設定

```bash
# crontab -e で設定

# ===== 日次: 毎日 7:00 JST（米市場終了後 = EST 17:00 + 9h）=====
# 対象: 金利, スプレッド, SRF, 失業保険(3年分修正検知), Yahoo市場指標(14日)
0 7 * * * cd /path/to/株式AI && python app/batch/run.py --daily >> logs/batch_daily.log 2>&1

# ===== 週次: 毎週土曜 8:00 JST（FRED 週次データ更新後）=====
# 対象: FRBバランスシート, SRF, MMF, 雇用統計(月次修正検知) + Layer再計算
0 8 * * 6 cd /path/to/株式AI && python app/batch/run.py --weekly >> logs/batch_weekly.log 2>&1

# ===== 月次: 毎月1日 9:00 JST（全データ3年分 + 修正検知 + Layer再計算）=====
0 9 1 * * cd /path/to/株式AI && python app/batch/run.py >> logs/batch_monthly.log 2>&1
```

### なぜこの時間?

| 頻度 | 時間 (JST) | 理由 |
|------|-----------|------|
| 日次 | 7:00 | 米市場16:00 EST 終了 → FRED/Yahoo 更新完了後 |
| 週次 | 土 8:00 | FRED 週次系列は木〜金に更新、土曜に安全に取得 |
| 月次 | 1日 9:00 | 前月のFRED改定データが反映される時期 |

## 修正検知（上方修正/下方修正）

### 仕組み

upsert 前に、新データと既存データ（Supabase上）を比較。値が変わっていたら `data_revisions` テーブルに自動記録。

```
新データ取得 → 既存データと比較 → 差異あり → data_revisions に INSERT → upsert で上書き
```

### 監視対象テーブルとカラム

| テーブル | 監視カラム | 修正が起きるケース |
|----------|-----------|-------------------|
| `fed_balance_sheet` | soma_assets, rrp, tga, reserves | FRED 速報→改定（特に reserves は月次改定あり） |
| `interest_rates` | fed_funds, treasury_2y, treasury_10y | FFレートは月次改定、国債利回りはまれ |
| `credit_spreads` | hy_spread, ig_spread | BofA 指数改定（まれ） |
| `mmf_assets` | total_assets | 四半期データの改定 |
| `weekly_claims` | initial_claims, continued_claims | 毎週1,000〜5,000件の修正が通常 |
| `economic_indicators` | current_value, u3_rate, u6_rate, nfp_change | BLS月次改定（過去2ヶ月分）+ 年次ベンチマーク改定 |

> Yahoo Finance / NY Fed のデータは確定値なので監視対象外。

### data_revisions テーブル

```sql
CREATE TABLE data_revisions (
    id            SERIAL PRIMARY KEY,
    table_name    TEXT NOT NULL,           -- 'fed_balance_sheet' 等
    record_date   DATE NOT NULL,           -- 修正されたレコードの日付
    column_name   TEXT NOT NULL,           -- 'soma_assets' 等
    old_value     DECIMAL,
    new_value     DECIMAL,
    change_amount DECIMAL,                 -- new - old
    change_pct    DECIMAL,                 -- 変化率(%)
    direction     TEXT,                    -- '上方修正' or '下方修正'
    detected_at   TIMESTAMPTZ DEFAULT NOW(),
    batch_run_id  TEXT                     -- 同じバッチ実行でグループ化
);
```

**SQL ファイル**: `app/batch/sql/create_data_revisions.sql`

Supabase の SQL Editor にコピペして実行するだけ。

### ログ出力例

修正が検知されると WARNING レベルでログに出力:

```
修正検知: 3件 (上方修正=1, 下方修正=2)
  fed_balance_sheet.reserves [2025-12-01]: 3.2145 → 3.2089 (下方修正 -0.1742%)
  interest_rates.fed_funds [2025-11-01]: 4.58 → 4.57 (下方修正 -0.2183%)
  fed_balance_sheet.soma_assets [2025-12-04]: 6.8432 → 6.8501 (上方修正 0.1009%)
```

### 取得範囲が3年の理由

FRED/BLS のデータ改定は複数の時間スケールで発生する:

| 改定タイプ | 対象 | タイミング | 例 |
|-----------|------|----------|-----|
| 月次改定 | NFP（過去2ヶ月分） | 毎月第1金曜 | 1月速報値が3月に改定 |
| 週次改定 | 失業保険申請 | 毎週木曜 | 先週値が1,000件修正 |
| 年次ベンチマーク | NFP（過去1年分） | 毎年2月 | 2025年2月: -58.9万人 |
| 季節調整改定 | 大半の系列 | 毎年1月 | 過去5年分を遡及修正 |

3年分（1095日）取得すれば年次ベンチマーク改定の影響範囲をほぼ全てカバーできる。

### 4. 手動データ（POST API で入力）

以下のデータは公開APIがない、または FRED の更新が遅いため手動入力が必要。

---

#### margin_debt（FINRA 信用取引残高）

**エンドポイント**: `POST /api/liquidity/margin-debt`

**データ取得先**: https://www.finra.org/investors/learn-to-invest/advanced-investing/margin-statistics

**更新頻度**: 月次（毎月下旬に前月分が公開）

**入力例**:
```json
{
  "date": "2026-01-01",
  "debit_balance": 935000,
  "free_credit": 185000
}
```

> 単位: FINRA サイトの百万ドルをそのまま入力。API 内部でドル変換 + 2年前比(change_2y)を自動計算。

---

#### ADP 雇用統計

**エンドポイント**: `POST /api/employment/indicators`

**データ取得先**: https://adpemploymentreport.com/

**更新頻度**: 月次（NFP発表の2日前、水曜）

**なぜ手動?**: FRED にも ADPWNUSNERSA があるが更新が遅れることがある

**入力例**:
```json
{
  "indicator": "ADP",
  "reference_period": "2026-01-01",
  "current_value": 183,
  "notes": "1月ADP: +183千人"
}
```

---

#### Challenger 解雇発表

**エンドポイント**: `POST /api/employment/indicators`

**データ取得先**: https://www.challengergray.com/press-releases/ (月次プレスリリース)

**更新頻度**: 月次（毎月上旬に前月分を発表）

**なぜ手動?**: FRED に系列が存在しない

**入力例**:
```json
{
  "indicator": "CHALLENGER",
  "reference_period": "2026-01-01",
  "current_value": 34817,
  "notes": "1月解雇発表: 34,817件"
}
```

---

#### NFP 改定メモ

**エンドポイント**: `POST /api/employment/indicators`

**なぜ手動?**: BLS が発表する改定値の解釈は人間の判断が必要（例: ベンチマーク改定で-58.9万人）

**入力例**:
```json
{
  "indicator": "NFP",
  "reference_period": "2025-03-01",
  "current_value": 157466,
  "nfp_change": 228,
  "notes": "2025年2月ベンチマーク改定後の確定値"
}
```

> NFP の速報値は `--employment` バッチで自動取得済み。改定時に手動で上書きし、`economic_indicator_revisions` に改定履歴が自動記録される。

### 手動データまとめ

| データ | 取得先URL | 頻度 | 投入方法 |
|--------|----------|------|----------|
| FINRA 信用取引残高 | [FINRA Margin Statistics](https://www.finra.org/investors/learn-to-invest/advanced-investing/margin-statistics) | 月次（下旬） | `POST /api/liquidity/margin-debt` |

### 毎月やること（手動入力チェックリスト）

1. **ADP雇用統計**（毎月第1水曜頃）
   - https://adpemploymentreport.com/ から民間雇用変化（千人）を取得
   - `python app/batch/manual_input.py add ADP 2026-02 XXX`
   - ※過去データ（2010〜）は `load-adp` で一括投入済み。新月分のみ追加

2. **Challenger人員削減**（毎月第1木曜頃）
   - https://www.challengergray.com/press-releases/ からレイオフ発表件数を取得
   - `python app/batch/manual_input.py add CHALLENGER 2026-02 XXXXX`

3. **Truflationインフレ率**（任意）
   - https://truflation.com/ から代替インフレ率（%）を取得
   - `python app/batch/manual_input.py add TRUFLATION 2026-02 X.X`

4. **FINRA信用取引残高**（毎月下旬）
   - https://www.finra.org/ から取得
   - `curl -X POST .../api/liquidity/margin-debt -d '{"date":"2026-02-01","debit_balance":935000}'`

## Supabase 書き込み (`db.py`)

### upsert メカニズム

1. **修正検知**: 対象テーブルの場合、既存データを取得して新データと比較
2. **差異記録**: 値が変わっていたら `data_revisions` に INSERT
3. **upsert**: 500行ずつバッチ upsert（`ON CONFLICT` で上書き）
4. `updated_at` タイムスタンプを自動付与

### ページネーション（読み取り時）

Supabase REST API はデフォルトで1000行の取得上限。
`calculators/layer_stress.py` の `_fetch_all()` で `.range()` を使ったページネーションにより全行取得。

## Layer Stress 計算 (`calculators/layer_stress.py`)

### 処理フロー

1. **Prefetch**: 8テーブルからページネーション付きで全データ取得
2. **月次マップ構築**: 各テーブルのデータを YYYY-MM 単位で最終行をマップ化
3. **月次ループ**: 対象月ごとに Layer1〜2B + Credit Pressure + Market State を計算
4. **Upsert**: `layer_stress_history` と `market_state_history` に書き込み

### Layer 構成

| Layer | 計算関数 | 入力データ | 意味 |
|-------|----------|-----------|------|
| Layer 1 | `calculate_layer1_stress()` | Net Liquidity (SOMA - RRP - TGA)、履歴Z-score | FRB流動性の構造的ストレス |
| Layer 2A | `calculate_layer2a_stress()` | 準備預金MoM、KRE 52W変化、SRF利用、IGスプレッド | 銀行システムストレス |
| Layer 2B | `calculate_layer2b_stress()` | 信用取引2Y変化、MMF 3M変化、VIX | リスクテイク/投機ストレス |
| Credit | `calculate_credit_pressure()` | HY/IGスプレッド、イールドカーブ、DXY | クレジット圧力（Low/Medium/High） |
| State | `determine_market_state()` | L1/L2A/L2Bスコア、L2A解釈タイプ | 市場状態コード判定 |

### 出力テーブル

#### layer_stress_history

| カラム | 型 | 説明 |
|--------|-----|------|
| `date` | text | 対象月末日 |
| `layer` | text | "layer1" / "layer2a" / "layer2b" |
| `stress_score` | float | ストレススコア (0-100) |
| `components` | jsonb | 計算に使った各コンポーネント値 |

#### market_state_history

| カラム | 型 | 説明 |
|--------|-----|------|
| `date` | text | 対象月末日 |
| `state` | text | 市場状態コード |
| `layer1_stress` | float | Layer 1 スコア |
| `layer2a_stress` | float | Layer 2A スコア |
| `layer2b_stress` | float | Layer 2B スコア |
| `credit_pressure` | text | "Low" / "Medium" / "High" |
| `comment` | text | 状態の説明テキスト |

### 手動入力テーブル

#### manual_inputs（乖離スコア用の手動データ）

```sql
CREATE TABLE manual_inputs (
  id SERIAL PRIMARY KEY,
  metric TEXT NOT NULL,          -- 'ADP_CHANGE', 'CHALLENGER_CUTS', 'TRUFLATION'
  reference_date DATE NOT NULL,
  value FLOAT NOT NULL,
  notes TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(metric, reference_date)
);
ALTER TABLE manual_inputs DISABLE ROW LEVEL SECURITY;
```

| metric | 値の単位 | 説明 |
|--------|---------|------|
| ADP_CHANGE | 千人（MoM変化） | ADP民間雇用変化。CSVからの一括投入で水準→変化に自動変換 |
| CHALLENGER_CUTS | 件数 | Challenger社のレイオフ発表件数 |
| TRUFLATION | % | Truflation代替インフレ率 |

> **RLS無効化が必須**: Railway バックエンドは `SUPABASE_ANON_KEY` を使うため、RLS が有効だとデータを読めない。

## 運用ガイド

### 初回セットアップ

```bash
# 1. .env にキーを設定（株式AI/.env）
# 2. Supabase で data_revisions テーブルを作成
#    → app/batch/sql/create_data_revisions.sql を SQL Editor で実行
# 3. 15年分のデータを一括取得
python app/batch/run.py --full --verbose
```

### 定期実行

cron 設定（上記「スケジューリング」セクション参照）を入れたら自動で動く。

### データ修正後の再計算

手動で生データテーブルを修正した場合:

```bash
python app/batch/run.py --calc
```

### 修正履歴の確認

Supabase Dashboard または API で:

```sql
-- 最近の修正一覧
SELECT * FROM data_revisions ORDER BY detected_at DESC LIMIT 20;

-- 特定テーブルの下方修正のみ
SELECT * FROM data_revisions
WHERE table_name = 'fed_balance_sheet' AND direction = '下方修正'
ORDER BY detected_at DESC;
```

### トラブルシューティング

| 症状 | 原因 | 対処 |
|------|------|------|
| `KeyError: 'SUPABASE_URL'` | `.env` 未設定 | 株式AI/.env にキーを追加 |
| `ModuleNotFoundError: dotenv` | パッケージ未インストール | `pip install python-dotenv` |
| FRED で 0件 | API キー無効 or 系列ID変更 | FRED サイトで系列確認 |
| Yahoo で 0件 | ティッカー廃止 or 市場休場 | yfinance 直接確認 |
| Layer計算で月数が少ない | Supabase 1000行上限 | `_fetch_all()` のページネーションで対応済み |
| 書き込みエラー | RLS ポリシー or カラム不一致 | service_role キーを使用 / スキーマ確認 |
