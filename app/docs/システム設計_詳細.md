## 3. データベース設計

### 3.1 設計方針

**新システムに持ち込むタブ:**
- 統合タブ
- 配管タブ
- 米国景気警戒タブ
- シグナルタブ
- 保有タブ

**保存するもの:**
- 銘柄マスター（カテゴリ、名前等）
- 流動性データ（FRBバランスシート、金利、クレジットスプレッド等）
- 経済指標（修正履歴対応・縦持ち方式で無限追加可能）
- 市場状態履歴（配管・統合タブ用）
- 個人データ（保有・取引・ウォッチリスト・設定）
- 一時キャッシュ（5分TTL、複数ユーザー共有）

**保存しないもの:**
- 株価OHLCV → yfinanceで都度取得
- リアルタイム価格 → yfinanceで都度取得
- EMA/ATR等 → 株価から都度計算
- シグナル結果 → Railwayで都度計算（DB保存不要）

## 4. キャッシュ戦略

### 4.1 なぜキャッシュが必要か

```
問題: yfinanceにはレート制限がある

ユーザーA → NVDAを見る → yfinance API
ユーザーB → NVDAを見る → yfinance API  ← 無駄！

解決: 一時キャッシュ（5分TTL）

ユーザーA → NVDAを見る → yfinance API → DBに一時保存
ユーザーB → NVDAを見る → DBから返す（API呼ばない）
5分後 → 自動削除
```

### 4.2 キャッシュのライフサイクル

```
1. ユーザーが銘柄を要求
2. DBのstock_cacheを確認
   ├── あり＆有効期限内 → DBから返す
   └── なし or 期限切れ → yfinance API呼び出し → DBに保存 → 返す
3. 定期的に期限切れを削除（cron or GitHub Actions）
```

### 4.3 キャッシュクリーンアップ

```sql
-- 5分ごとに実行
DELETE FROM stock_cache WHERE expires_at < NOW();
```

3. Include → Emails にメールアドレスを追加 → Save

### 12.4 CORS設定

Backend（`app/backend/main.py`）で `https://open-regime.pages.dev` を許可済み。

### 12.5 Git連携について

GitHub リポジトリ名 `-` が Cloudflare Pages のGit連携で認識されないため、手動デプロイを採用。
リポジトリ名を変更すればGit連携（`git push` で自動デプロイ）が可能。

### 12.6 注意事項

- フロントエンド更新時は `wrangler pages deploy` が必要（Backendは `git push` で自動）
- Cloudflare Access の設定変更は Zero Trust ダッシュボードから（https://one.dash.cloudflare.com/）
- Pages Settings の「Access policy」はプレビューデプロイメントのみ対象。本番保護は Zero Trust 側で設定
